Exceptions vs. Runtime Failures

Beginners often ask: "Are exceptions and runtime errors the same thing?" The answer is nuanced. Let's break down the truth about when things go wrong.

Are all exceptions runtime events?

YES. By definition, an Exception only exists once the program is running. Even "Checked Exceptions" (which the compiler makes you write handlers for) only actually happen when the program is live.

Is everything that fails at runtime an Exception?

NO. "Runtime Failure" is a broad category. In Java, things that go wrong at runtime are split into Exceptions (fixable) and Errors (fatal system failures).

The "When" and "What" of Failures

Pre-Runtime
Compilation Errors

Syntax mistakes (missing ;). The program never even starts.

Runtime (Handled)
Exceptions

Logical or external issues (bad input). We catch these to stay alive.

Runtime (Fatal)
Errors

System failures (No Memory). We fail fast because the environment is broken.

Why don't we catch "Errors"?

If your program runs out of memory (OutOfMemoryError), there is no memory left to even run your "catch" code safely.

Action: Fail Fast Let it crash. It's better to stop than to run in a broken state.
Action: Fix External Solve it by changing JVM settings or fixing hardware, not by writing code.

Key Takeaway:

Every Exception is a runtime event, but not every runtime failure is an Exception. We use code to handle Exceptions, but we use system changes to address Errors.

4

Encounter: Input Mismatch

Scenario:

Your code expects an int, but the user types "ABC".

Scanner sc = new Scanner(System.in);
System.out.print("Enter an integer: ");
int x = sc.nextInt(); // Crashes here!

The Crash:

Exception in thread "main"
java.util.InputMismatchException

Simulator: Input "ABC" into a numeric field.

Key Takeaway:

Users are unpredictable. The InputMismatchException occurs when the Scanner's retrieved data type doesn't match the expected type.

5

Encounter: Divide by Zero

Scenario:

A user inputs "0" as a divisor. Java cannot perform this calculation.

int divisor = sc.nextInt();
double result = 100 / divisor; // Exception if 0!

The Result:

java.lang.ArithmeticException: / by zero

Simulator: Try dividing 100 by 0.

Key Takeaway:

Logical errors (like dividing by zero) can trigger exceptions even if the code syntax is perfectly correct.

7

Handle, Catch, Throw

Claiming (Throws)

void riskyMethod() throws Exception {
  if (badStuff) 
    throw new Exception();
}

"I might fail, whoever calls me needs to be ready."

Catching (Try-Catch)

try {
  riskyMethod();
} catch (Exception e) {
  System.out.println("Handled!");
}

"I'll try this, and if it fails, I have a backup plan."

Key Takeaway:

Methods use "throws" to claim responsibility and "try-catch" blocks to wrap risky code with a recovery plan.

API Knowledge & Logic

How to know what to catch?

Always check the Java API documentation. The method signature tells you everything:

public String readLine() throws IOException MUST BE CAUGHT

This is a Checked Exception. The compiler will complain if you ignore it.

public static int parseInt(String s) OPTIONAL CATCH

This throws a NumberFormatException (Unchecked). Handle it only if you expect bad user strings.

Key Takeaway:

Use Checked exceptions for external risks beyond your control (files, networks). Use Unchecked exceptions for logic errors you should have prevented.
