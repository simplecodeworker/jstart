Section 1

Introduction to Static Methods

Key Concept: Instance vs. Static Context

The Core Concept

So far, every method you've written inside a class could only be run after building an actual object with the new keyword. But what if you need to run a task before any objects exist?

Java provides Static Methods. By adding the modifier word static, the method is bound directly to the class blueprint itself, rather than individual physical copies (objects).

💡 Why is main() static?

public static void main(String[] args)
The JVM must boot your code directly. If main wasn't static, Java would face a paradox: it would need to instantiate an object of your class just to find the start button!

Code Blueprint & Syntax

Java Sample 
// Calling a utility static method without creating an object first!
double result = Math.sqrt(4.5);
System.out.println(result);

Key Takeaway: Use static methods when you need functionality that belongs to the class as a whole (like pure mathematical formulas) rather than to any specific instance.

Section 2

Accessing Static Methods & Variables

Key Concept: Shared State (UnitsLeft)

Shared Class State

When a variable is declared as static, it behaves as a single shared storage location for the entire class. If one instance modifies it, all other instances observe the modified value immediately!

Consider the SPApartmentRental class. Every time we rent an apartment, we decrement the global variable UnitsLeft.

⭐ Invocation Syntaxes (Preferred Method)

  • SPApartmentRental.getUnitsLeft() (Preferred: explicit, does not hide class-level mechanics)
  • s1.getUnitsLeft() (Syntactically legal, but bad style because it disguises static fields as instance fields)
SPApartmentRental.java Complete Code 
class SPApartmentRental {
    private double rental;
    private static int UnitsLeft = 10; // Static Variable

    SPApartmentRental(double r) {
        UnitsLeft--; // Decrements global shared counter
        rental = r;
    }

    static int getUnitsLeft() { // Static Method
        return UnitsLeft;
    }
}

public class TestStaticMethod {
    public static void main(String[] args) {
        System.out.println(SPApartmentRental.getUnitsLeft()); // prints 10
        SPApartmentRental s1 = new SPApartmentRental(1800.50);
        System.out.println(s1.getUnitsLeft()); // prints 9
    }
}

Section 3

The Rules and Scope of Static Methods

Key Concept: Static-to-Static Only

The "Static Context" Ruleset

Because static methods are initialized when the application launches (before any objects have been constructed), they have strict interaction restrictions.

The key restriction is the Static-to-Static Rule:

⚠️ Fatal Compiler Restriction

A static method can call ONLY other static methods and static variables. If you attempt to access an instance field directly, the compiler halts build production with:
"non-static variable cannot be referenced from a static context"

Scope Checklist Reference

✔ Class_name.method_name() (Valid Direct Access)
✔ Can read any other Static Variable in scope
✘ No reference to this keyword exists
✘ Cannot call non-static instance methods directly

Section 4

Invalid Non-Static References

Key Concept: Intentional Compilation Bugs

Code Alteration Breakdown

To truly grasp the static concept, let us intentionally break our previous SPApartmentRental class example.

Imagine removing the word static from the declaration of UnitsLeft, changing it into a plain instance field. Here's what goes wrong:

❌ The Critical Fail Sequence:
  1. JVM reaches line SPApartmentRental.getUnitsLeft().
  2. The class blueprint exists, but absolutely no rental objects have been made yet.
  3. The static method attempts to load UnitsLeft.
  4. Because it is now an instance variable, it requires a specific object's memory location to read.
  5. Crash! The compiler prevents this contradiction entirely.
BUG INDUCED CODE
BrokenSPApartmentRental.java 
class SPApartmentRental {
    private double rental;
    private int UnitsLeft = 10; // ERROR: Non-static!

    SPApartmentRental(double r) {
        UnitsLeft--;
        rental = r;
    }

    static int getUnitsLeft() {
        return UnitsLeft; // COMPILER ERROR DETECTED HERE
    }
}

Section 5

Mechanisms of Passing Variables

Key Concept: Value vs. Reference Pointer

Primitive Memory Routing

When you pass variables to Java methods, how Java manages the hardware memory depends entirely on whether the variable is a Primitive Type or an Object Reference Type.

🎯 Passed by Value (Primitives)

Includes basic types: int, double, float, boolean, char.
An entirely independent copy of the data value is created inside the method's local stack block. Changes inside the method never propagate back to the caller environment!

Object Reference Handling

When passing complex entities like Object variables, custom structures, Arrays, or Strings, Java copies the reference pointer address instead of copying the massive object payload.

🔗 Passed by Reference Pointer

No separate copy of the variable values is generated on the heap.
The method parameters reference the identical system memory slot as the caller. Modifying properties inside the method mutates the object permanently.

Section 6

Passing Objects by Reference Pointer

Key Concept: Persistent Mutations

Code Visualization: Animal Class

To verify this reference sharing mechanism, let us trace an interactive simulation using our Animal leg chopper program.

We create a caterpillar with 16 legs. Then, we pass it into method ChopLeg(caterpillar, 2).

🔍 Persistent Mutations: Because the parameter variable X copies the pointer to the exact same heap coordinates, X.LegCount = X.LegCount - y changes the caterpillar object itself!
TestPassByRef.java 
class Animal {
    int LegCount;
    int age;
    Animal(int LC, int a) { LegCount=LC; age=a; }
}

public class TestPassByRef {
    public static void main(String[] args) {
        Animal caterpillar = new Animal(16, 1);
        ChopLeg(caterpillar, 2);
        System.out.println("Legs left:" + caterpillar.LegCount);
        // Terminal output: Legs left: 14
    }

    static void ChopLeg(Animal X, int y) {
        X.LegCount = X.LegCount - y;
    }
}

Section 7

Passing Primitives by Value

Key Concept: Value Isolation

Primitive Isolation Demonstration

To contrast with Section 6, let's explore what happens when we pass a primitive integer int instead of an object.

We initialize int i = 100; inside main. We then call the method ABC(i), which does y = y + 100; inside its body.

💡 What Actually Happens: The value 100 is copied into local variable y. When y becomes 200, it stays completely isolated within the stack frame of ABC(). The original i back in the main scope remains unchanged at 100!
TestPassByValue.java 
public class TestPassByValue {
    public static void main(String[] args) {
        int i = 100;
        ABC(i);
        System.out.println("i : " + i);
        // Terminal output: i : 100
    }

    static void ABC(int y) {
        y = y + 100;
    }
}
