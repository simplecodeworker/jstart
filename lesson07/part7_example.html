Understanding Variables Scope in Java

In Java, where you declare a variable determines its scope—meaning where in your program that variable can be read, written, and recognized. There are two primary scopes demonstrated in this program:

1. Instance Variables (Fields)

Declared directly inside a Class but outside any method. They belong to the Object (stored in the Heap memory). Every method inside the class has access to them. They persist as long as the object exists.

2. Local Variables

Declared inside a Method block { ... }. They are temporary, stored in the Stack memory, and are destroyed once the method finishes. They cannot be seen outside their parent method.

Method 1: Shadowing the Gold Variables

How local variables block instance variables with identical names

TestVariableScope.java Method A Scope 
  void dCPE3APP() {
    // Local variable for method dCPE3APP()
    int noOfGold = 2; // This SHADOWS the instance variable!

    System.out.println("Class DCPE3APP got " + noOfGold + " gold medals");
    // Outputs: 2 (Uses Local)

    System.out.println("There are altogether " + noOfSilver + " silver medals");
    // Outputs: 6 (No local duplicate, looks up to Instance)
  }
Scope Mechanics

Variable Shadowing Explained

Inside dCPE3APP(), we declare a local variable named noOfGold and set it to 2. Because this variable matches the name of our instance field (noOfGold = 5), it temporarily shadows or conceals the instance field within this block.

  • ⚠ Any reference to noOfGold inside this method resolves to 2 (the local copy).
  • ✔ For noOfSilver, Java checks the local scope, doesn't find a local version, and automatically retrieves the instance field's value (6).

Method 2: Clean, Isolated Local Scope

How separate methods maintain their own private local states

TestVariableScope.java Method B Scope 
  void dCPE3BQQ() {
    // Local variable for method dCPE3BQQ()
    int noOfGold = 3; // Isolated: totally separate from dCPE3APP!

    System.out.println("Class DCPE3BQQ got " + noOfGold + " gold medals");
    // Outputs: 3 (Uses Local)

    System.out.println("There are altogether " + noOfSilver + " silver medals");
    // Outputs: 6 (Accesses same instance variable)
  }
Method Boundaries

Scope Encapsulation

The method dCPE3BQQ() resides in its own isolated stack bubble. Even though both methods define a variable named noOfGold, they cannot see each other’s variables.

  • ✔ The local variable noOfGold = 2 inside dCPE3APP() is already destroyed or completely inaccessible here.
  • ✔ This method introduces its own local noOfGold = 3, shadowing the instance variable independently.

The Application Code (The Entry Point)

Instantiating objects and tracing the program output

TestVariableScope.java Application Main 
  public static void main(String[] a) {
    // 1. Create instance (the Object)
    TestVariableScope medals = new TestVariableScope();

    // 2. Invoke method A
    medals.dCPE3APP(); 

    // 3. Invoke method B
    medals.dCPE3BQQ(); 
    
    // 4. Direct access of the instance field from main
    System.out.println("There are altogether " + medals.noOfGold + " gold medals");
    // Outputs: 5 (Because direct access queries the instance field!)
  }
Execution Flow

Running the Application

The main method is the driver of the application. Notice step 4: when we access medals.noOfGold, we do so by explicitly referencing the object variable (medals).

  • ✔ The local variables initialized inside dCPE3APP and dCPE3BQQ are completely dead at this point.
  • ✔ Thus, accessing medals.noOfGold gets the original, untouched instance value of 5.

Java Variable Scope Rule Book for Beginners

1. Declaration Zone determines Scope

Where you write type name = value is the blueprint. Variables declared outside methods are Instance variables, inside are local.

2. Shadowing Blocks but Doesn't Destroy

Declaring a local variable with the identical name as an instance variable hides the instance variable. But the instance variable value on the object remains unchanged!

3. Life Span (Garbage Collection)

Local variables disappear as soon as the method ends. Instance variables persist in Heap memory as long as your object instance (medals) remains alive in code.
