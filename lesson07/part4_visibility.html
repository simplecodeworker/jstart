The Core Concepts

SECTION 1: The Four Java Visibility Modifiers and Their Targets

Visibility (or access) modifiers are the keywords in Java that declare the accessibility scope of classes, variables, and methods. They act as protective boundaries to control data leakage and promote encapsulation.

Target Rules for Modifiers:

  • • For Classes: You can only use public or no modifier (the default/package-private) for outer class declarations.
  • • For Variables & Methods: You can apply all four modifiers: public, protected, private, or no modifier.

Modifier Definitions

PUBLIC:

Visible everywhere in the entire project.

PROTECTED:

Visible within subclasses/child classes or files of the current package.

NO MODIFIER (Default):

Known as "package-private". Visible exclusively in the same package.

PRIVATE:

Visible only in the current class. Instantiated objects cannot access directly.

The Rules Matrix

SECTION 2: The Access Control Matrix

To easily track visibility scopes, Java developers use an **Access Levels Matrix**. There are four primary contexts from which an access attempt can originate:

Class Same class definition file
Package Classes in the same package folder
Subclass/Child class Classes extending the owner from anywhere
World Any unrelated class, anywhere

Boundary Testing

SECTION 4: Cross-Package Access and the Limits of Public Visibility

When files belong to distinct packages, the project compile boundaries are actively tested. Classes must explicitly import target scopes. Let's study how FirstPackage interacts with SecondPackage.

Java Project Structure
📁 src
📁 FirstPackage
📄 Package1Main.java
📁 SecondPackage
📄 Package2Class1.java
The Cross-Package Access Constraint

Since Package1Main resides inside FirstPackage, it has absolutely **zero access** to anything declared inside SecondPackage *unless* that member is explicitly declared public. Any attempt to reference default, protected, or private variables will fail compiler authorization.

Local Interoperability

SECTION 5: Intra-Package Access and Component Interaction

In contrast to cross-package systems, classes stored inside the **exact same folder** packages interact in a highly collaborative way. This is known as **Intra-Package Access**. Let's examine how Package2Class1 and Package2Class2 communicate.

Intra-Package Filesystem
📁 src
📁 SecondPackage
📄 Package2Class1.java
📄 Package2Class2.java
Package-Level Accessibility Rules

Since Package2Class2 resides in the exact same package directory, it easily receives access to variables marked as public, protected, and default (no modifier). The only state boundary it is completely blocked from accessing is private, which remains locked inside the declaring class template.
