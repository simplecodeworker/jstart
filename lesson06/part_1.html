The Procedural Drawback

Cluster Paradox

In Procedural Programming, information is clustered around Attribute Types (All names together) and NOT the individual account holder.

❌ Fragmentation: The "Account" unit doesn't exist.
❌ Mental Overhead: Programmer must manually sync indices.

Data is "Horizontal" (By Type)
instead of "Vertical" (By Person)

The Method Signature Mess

In procedural code, methods are separate from data. You must pass all arrays into every function.

// Procedural "Withdraw" requires passing the whole balance array
double withdraw(double[] balanceArray, double amt, int accountIndex) {
  balanceArray[accountIndex] -= amt;
  return balanceArray[accountIndex];
}
Notice: The logic has no "owner". Any part of the program can call this and potentially mess up the indices.

11

Implementation: BankAC

PascalCase: Class names start with an Uppercase letter.
No 'static': These belong to individual instances.
class BankAC {
  // Attributes (Properties)
  String acNo;
  String name;
  double balance;

  // Methods (Behaviors)
  public void deposit(double amt) {
    balance += amt;
  }

  public boolean withdraw(double amt) {
    if (balance >= amt) {
      balance -= amt;
      return true;
    } else return false;
  }
}

Definition vs. Execution

The Blueprint (Class)

class BankAC { ... attributes and methods ... }

The Execution (Main Method)

public class TestBank {
  public static void main(...) {
    BankAC b1 = new BankAC();
  }
}
Pro Tip: The 'driver' class is technically a "client" of your BankAC class. They can even live in separate files!
