Section 1

Manual Initialization

Before learning constructors, we must understand how objects are created "the hard way." This section introduces the basic BankAC class and shows how to set values manually.

The Class Structure

class BankAC {
    String acNo;
    String name;
    double balance;
    
    public void deposit(double amt){
        balance += amt;
    }
    // ... withdraw method
}

Manual Setup (The Driver)

BankAC b1 = new BankAC(); 
b1.name = "Jane Fonda";
b1.acNo = "112233";
b1.balance = 1000.34;

Key Takeaways:

  • Manual Initialization: Without a constructor, you must set every attribute manually using dot notation.
  • Default Constructor: Java provides a hidden "default" constructor that allows new BankAC() even if you didn't write one.
  • Inefficiency: This is tedious and error-prone for many objects.

Section 2

The Constructor Concept

A constructor is a special method used to initialize objects.

Same Name

Matches the Class name exactly

No Return

No return type, not even void

Purpose

Constructs and initializes state

Auto-Invoke

Triggered by the 'new' keyword

Clear Takeaway:

Constructors are the blueprint's "setup instructions." They ensure every object starts with valid data immediately upon birth.

Section 3

Implementation & Invocation

Implementing the Constructor

public class BankAC {
    String acNo;
    String name;
    double balance;

    // Constructor acts as a gateway
    public BankAC(String ac, String n, double b) {
        acNo = ac;
        name = n;
        balance = b;
    }
}
Usage In Driver:
new BankAC("112233", "Jane", 1000.34);
Invocation result:
  • acNo gets "112233"
  • name gets "Jane"
  • balance gets 1000.34

Section 4

Independence & Reusability

One class can create thousands of independent objects. Each object lives in its own space in memory with its own unique data.

Simulator: Memory Layout

Object Instance: b1
name: "Jane"
acNo: "112233"
bal: 1000.34
Object Instance: b2
name: "John"
acNo: "445566"
bal: 10.00

Learning Point:

  • Object Independence: Every instance maintains separate data. Changing b1 does not affect b2.
  • Reusability: We write the constructor once but use it to scale our app infinitely.

Section 5

The "this" Keyword

What happens when the parameter name is the same as the attribute name? We get Shadowing. The this keyword helps us solve it.

Disambiguation Solution:
public BankAC(String name, double balance) {
    this.name = name;        // this.name is the Class Variable
    this.balance = balance;  // balance is the incoming Parameter
}
Shadowing

Parameters "hide" the attributes because they share the same name.

Dynamic Reference

this refers to the specific instance currently being constructed (b1, b2, etc.).

Quiz: In the line 'this.age = age', what does 'this.age' refer to?

Section 6

Flexibility & Default Values

A constructor doesn't have to take every attribute as a parameter. You can hardcode default logic directly inside the constructor.

Simplified Constructor
public BankAC(String ac, String n) {
    acNo = ac;
    name = n;
    balance = 0.0; // Hardcoded default for new accounts
}
Pro Tip:

This is extremely useful when some data is common for all new objects, like starting a game with zero score or starting an account with zero balance.

Section 7

Core OOP Concepts

Real-World Modeling

Objects represent physical or conceptual things (Cars, Bank Accounts, Users). Code reflects reality.

Encapsulation

Think of a "box" that hides data. We bundle attributes and behaviors together and protect them from outside interference.

Object Interaction

Object A (Sender)
Sends "Message" (Method Call)
Object B (Receiver)

Objects interact by invoking methods on one another.
