Section 1

Foundational OOP Terminology

Terminology Reference

OOP TermDefinition
methodSame as function/behaviour. To call a method: x.f(y), where x is an object of class that contains f method.
send a messageCall a method, invoke/use a method.
instantiateCreate object/instance from template class with new.
classA template/blueprint/structure with both data and functions. A user-defined complicated data type.
objectConcrete thing/instance created from class with new. Memory is allocated to it.
memberA data field or method is a member of a class if it's defined in that class.
constructorA special method that creates and initializes new objects. It has the same name as class name, no return data type, not even void. Can be overloaded.

Key Takeaways

  • Blueprint vs. Reality: A class is merely a structural template, whereas an object is a concrete instance of that template occupying physical memory.
  • State and Behavior: Classes are made of members, which include data fields (state) and methods (behavior).
  • Initialization: A constructor is a distinct, specialized block of code responsible purely for setting up a newly allocated object.

Section 2

Inheritance and Advanced Concepts

OOP TermDefinition
InheritanceDefining a class (child) in terms of another class (parent). All of the public members of the public class are available in the child class.
PolymorphismDefining functions with the same name, but different parameters. (Note: Specifically describing compile-time polymorphism/overloading here).
overloadA method is overloaded if there is more than one definition (same method name but different parameters in the same class).
overrideIn a child class, redefine a method from a parent class. (same method name and parameter, different implementation in child and parent class).
subclassSame as child, derived, or inherited class.
superclassSame as parent or base class.
attributeSame as data member/member field/property.

Key Takeaways

  • Hierarchies: Inheritance creates a parent-child relationship (superclass and subclass) allowing for massive code reuse.
  • Overloading vs. Overriding: Overloading happens within the same class (same name, different parameters). Overriding requires inheritance (redefining a parent's exact method signature in a child class).
  • Synonyms: Terms like attribute, data member, member field, and property are largely interchangeable.

Code Comparison: Overloading vs Overriding

OVERLOADING (Same Class)
class Calculator {
  // Method 1
  int add(int a, int b) {
    return a + b;
  }

  // Method 2 (Overloaded: Different params)
  int add(int a, int b, int c) {
    return a + b + c;
  }
}
OVERRIDING (Parent-Child)
class Animal { // Superclass
  void speak() {
    System.out.println("...");
  }
}

class Dog extends Animal { // Subclass
  @Override // Same exact signature
  void speak() {
    System.out.println("Woof!");
  }
}

Encapsulation and Java Modifiers

Modifiers determine visibility and behavior across classes, variables, methods, and constructors.

Modifierclassvariablemethodconstructor

Key Takeaways

  • Access Control: Visibility is strictly layered. public is accessible everywhere, protected limits access to packages/subclasses, default restricts to the package, and private locks access to the specific class only.
  • Structural Modifiers: final guarantees immutability (variables can't change, methods can't override, classes can't inherit). static binds a member to the class blueprint itself, rather than to object instances.
  • Applicability: Not all modifiers apply to all components (e.g., constructors cannot be static or final).

Section 4

Runtime Polymorphism

☕
  • Polymorphism allows the same method name to be used in conjunction with different objects.
  • The correct method chosen during running depends on the type of object it is associated with.

Key Takeaways

  • Flexibility: You can use a generic interface or parent class reference to interact with a variety of distinct child objects using the same method signature.
  • Runtime Resolution: The JVM handles mapping the method call to the correct overridden method implementation depending on the specific object type during execution (Dynamic Method Dispatch).
