SECTION 1: Lesson Objectives

This module sets the foundational agenda for understanding constructor chaining and invocation within an inheritance hierarchy in Java.

Key Learning Points and Takeaways

  • Constructors have a specific execution order in inheritance.
  • Java provides automatic mechanisms for constructor execution.
  • Developers can manually control constructor execution using specific keywords like super() and this().

SECTION 2: The Automatic super() Call and its Execution Sequence

The super() Call in Subclass

A subclass must interact with its parent's constructor to fully initialize an object.

  • 1 The super() keyword is specifically used for calling a parent class's constructor.
  • 2 If a programmer does not write a super() call, Java will automatically insert an invisible one that calls the parent's no-argument constructor.
  • 3 The hidden super() call is always placed as the very first line of the subclass constructor.
class Person {
Person() { System.out.println("Person's no-arg constructor"); }
}
class Employee extends Person {
Employee() {
// (hidden auto super() call)
Activates Person()
System.out.println("Employee's no-arg constructor");
}
}

SECTION 3: Behavior of Auto super() with Overloaded Constructors

A common point of confusion is what happens when you call an overloaded constructor in the subclass.

public class testSuperChain {
public static void main(String[] args) {
  // Calling overloaded constructor
  Employee e2 = new Employee(100);
}
}
class Person {
Person() { System.out.println("Person's no-arg constructor"); }
}
class Employee extends Person {
Employee() { ... } // Default
Employee(int a) {
// Auto super() call still goes to Person()
System.out.println("Employee's 2nd constructor");
}
}

Output Sequence

  1. Person's no-arg constructor
  2. Employee's 2nd constructor

Key Learning Points

  • Passing an argument to a subclass constructor does not mean those arguments are automatically passed up to the parent constructor.
  • Unless explicitly told otherwise, Java will always trigger the parent's no-arg constructor, regardless of which overloaded subclass constructor is executing.

SECTION 4: The Chain Effect in Multi-Level Inheritance

The rule of automatic constructor calling applies all the way up the inheritance tree. This creates a powerful Chain Effect.

Class Object (Base)
Class Person
Class Employee
Class Faculty
Execution resolves strictly top-down (Base first, Faculty last).
Instantiating Faculty: new Faculty()
class Faculty extends Employee {
  Faculty() {
    // 1. Hidden super() goes to Employee
    System.out.println("Faculty's...");
  }
class Employee extends Person {
  Employee() {
    // 2. Hidden super() goes to Person
    System.out.println("Employee's...");
  }
class Person {
  Person() {
    // 3. Hidden super() goes to Object!
    System.out.println("Person's...");
  }

Takeaway: Instantiating a subclass at the bottom of an inheritance hierarchy triggers a cascading chain of constructor calls all the way up to the highest superclass. The execution resolves from the top of the hierarchy downwards.

SECTION 5: Controlling Initialization with Explicit super() Calls

You can explicitly pass data to a parent class during initialization by overriding the hidden default call. This is done by writing an explicit super(...) statement.

Matching Signatures Table

Call from Subclass Matching Constructor in Superclass
super() Person()
super(int) Person(int)
super(int, String) Person(int, String)
Example
class Person {
Person() { System.out.println("Person's no-arg"); }
Person(int a) { System.out.println("Person's 2nd constructor"); }
}
class Employee extends Person {
Employee() {
  super(10); // Triggers matching Person(int a)
  System.out.println("Employee's no-arg constructor");
}
}

Crucial Rules for explicit super()

  • Writing an explicit super() call overrides and disables the compiler's hidden automatic super() call.
  • An explicit super() call must always be the very first statement inside the subclass constructor.
  • IMPORTANT: This applies only to the constructor

SECTION 6: Missing Implicit super() Constructor Error

This is one of the most common compilation errors beginners face in Java inheritance.

class Person {
// Only a parameterized constructor exists. Default no-arg is GONE.
Person(int a) { System.out.println("Person's only constructor"); }
}
class Employee extends Person {
Employee() {
// (hidden super() attempts to call Person() which doesn't exist!)
ERROR: Implicit super constructor Person() is undefined for default constructor.
  System.out.println("Employee's no-arg constructor");
}
}

The Cause and The Fix

If a parent class defines at least one constructor with arguments, Java does not provide a default no-argument constructor for it.

How to fix it?

The subclass must explicitly call the existing parameterized parent constructor using super(arguments); on the first line.

SECTION 7: Internal Chaining Using the this() Keyword

The this() Call

While super() talks to the parent class, this() talks to other constructors within the exact same class.

  • Used to call another constructor of its own class.
  • Like super(), it must be the 1st statement.
  • Using this() prevents the automatic super() call from being inserted into that specific constructor (the target constructor will handle the super call instead).
new Employee(4);
class Employee extends Person {
// Target Constructor
Employee() {
  // (Hidden auto super() runs here!)
  System.out.println("Employee's no-arg");
}
// Original Constructor
Employee(int a) {
  this(); // Jumps up to Employee()
  System.out.println("Employee's 2nd");
}
}

Execution Flow: When this() is called, the current constructor pauses, and the target constructor begins. The target constructor contains the implicit super() call to initialize the parent class first. The original constructor that invoked this() finishes its remaining statements last.
