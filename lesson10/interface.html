2. What is an Interface?

An interface is like a blueprint or a strict contract. It defines what a behavior is, but not how it is done. Let's compare it to a standard Java class.

Standard Class

Has variables (state) and concrete methods (behavior with logic inside {}).

void play() {
  System.out.println("Playing");
}
VS

Interface

Has abstract methods by default (method signature with no body, ends with ;).

void play();

Key Takeaways

  • Abstract methods define what needs to be done, not how.
  • While traditionally interfaces only held abstract methods, modern Java allows for default and static methods to provide fallback logic without breaking existing implementations.

3. Why Are Interfaces Relevant?

A common question from beginners is: "Why do I need an interface? I can just code the sendSOS() method directly inside the Phone class!"

While true for a solo programmer making a tiny app, real-world software is built by teams. Interfaces act as a binding contract between different developers to prevent integrated code from breaking.

Without Interface (Chaos)

Alice writes the Main program. Bob writes the Phone class.

  • Alice guesses Bob will name his method sendEmergency().
  • Bob names it sendSOS() instead.
  • Result: The code breaks when they combine it. Alice also had to wait for Bob to finish to know the final method names.

With Interface (Harmony)

They agree on an SMS interface first.

  • Alice writes her Main code using the agreed SMS interface rules.
  • Bob implements SMS, which forces him to use sendSOS().
  • Result: They work in parallel. The compiler guarantees Bob's code will perfectly match Alice's expectations.

Key Takeaways

  • Contracts: Interfaces guarantee that a class will have specific methods, eliminating guesswork between programmers.
  • Parallel Development: Teams can work independently on different parts of a program simultaneously.
  • Plug and Play: You can easily swap one implementing class for another (e.g., swapping an OldPhone for a SmartPhone) without breaking the rest of the program, as long as they share the same interface.

4. Define, Use and Test Interface

Let's look at the practical application. We define an interface, use a class to connect to it via the implements keyword, and write the actual logic.

Java
public interface SMS {
    public void sendSOS(String msg) ; // No details, ends with semicolon
}

public class Phone implements SMS {
    String phoneNumber;
    String dialNumber;
    
    public void sendSOS(String msg) { // Details of method provided here
        System.out.println("Sent " + msg + " to " + dialNumber);
    }
}

public class testInterface {
    public static void main(String[] args) {
        Phone p1 = new Phone();
        p1.dialNumber = "88888888";
        p1.sendSOS("Hello");
    }
}
Output: Sent Hello to 88888888

Key Takeaways

  • Create interfaces using the interface keyword.
  • A class connects to an interface using implements.
  • The implementing class must provide the actual logic (method body) inside {}.

5. Error Handling and Consistency

Implementing an interface is a strict contract. If you sign the contract (use implements), you must fulfill it by writing the logic for every abstract method. Let's see what happens if you forget.

public interface SMS {
    public void sendSOS(String msg) ;
}

public class Phone implements SMS {
    String phoneNumber;
    String dialNumber;
    
    // Oops! We forgot to write the sendSOS() method here!
}

Try compiling the code above:

> javac Phone.java
Error: Phone is not abstract and does not override abstract method sendSOS(String) in SMS
1 error

Key Takeaways

  • The Java compiler enforces consistency. Missing a method throws a compilation error.
  • This ensures any object claiming a specific behavior actually possesses the working logic for it.

6. Implementing Multiple Interfaces

Java classes can only extend one parent class. However, they can implement multiple interfaces simultaneously, separated by a comma. This is how Java allows multiple inheritance of behavior.

Interface SMS
Interface Call
Class Phone
interface SMS { public void sendSOS(String msg) ; }
interface Call { public void makeCall(int countryCode, String number) ; }

// Separated by a comma
class Phone implements SMS, Call {
    String dialNumber;
    
    public void sendSOS(String msg) {
        System.out.println ("Sent " + msg + " to " + dialNumber );
    }
    
    public void makeCall(int cc, String n) {
        System.out.println ("Calling +" + cc + "-" + n );
    }
}

Key Takeaways

  • Java restricts multiple inheritance of state (classes) to prevent ambiguity.
  • Java allows multiple inheritance of type/behavior (interfaces).
  • A single class can fulfill multiple distinct roles.

7. Combining Inheritance and Interfaces

A class can extend a parent class AND implement an interface at the same time. Interfaces allow completely unrelated objects (like an animal and a plant) to share a common behavior.

interface Edible { public String howToCook() ; }

class Animal { public String toString() { return ("Animal"); } }

// Extends MUST come before implements
class Chicken extends Animal implements Edible {
    public String howToCook() {
        return (" - Steam it with herbs.");
    }
    public String toString() { return ("chicken"); }
}

// A plant, unrelated to Animal, but still Edible
class WinterMelon implements Edible {
    public String howToCook() {
        return (" - Boil it with chicken feet.");
    }
    public String toString() { return ("winter melon"); }
}
Output:
chicken - Steam it with herbs.
winter melon - Boil it with chicken feet.

Key Takeaways

  • When combining them, the extends keyword must come before implements.
  • This demonstrates Polymorphism: An unrelated Chicken and WinterMelon can both be treated as Edible.

8. Interface Segregation Principle (ISP)

Interfaces are foundational architectural tools in enterprise applications. Writing an interface is easy, but designing a good one takes forethought.

"No client should be forced to depend on methods it does not use."

This is the core definition of ISP. "Clients" refers to the implementing classes. We want to avoid interface bloat—where an interface has too many unrelated methods.

Key Takeaways

  • ISP is a standard OOP best practice.
  • Don't force a class to implement behaviors it doesn't need.
  • Highly relevant in professional industry frameworks (e.g., Spring Framework).

9. A Bad Example (Violating ISP)

Let's look at a bloated Toy interface. A toy house cannot move or fly, but because it implements the bloated interface, the programmer is forced to create empty "dummy" methods.

Bad Design
interface Toy {
    void setPrice(double price);
    void setColor(String color);
    void move(); // Not all toys move
    void fly();  // Not all toys fly
}

class ToyHouse implements Toy {
    double price;
    String color;

    public void setPrice(double price) { this.price = price; }
    public void setColor(String color) { this.color=color; }

    // Dummy methods! Code Smell!
    public void move(){} 
    public void fly(){}
}

Key Takeaways

  • Code Smells: Empty method implementations are a huge red flag that ISP is violated.
  • Fragility: If we added walk() to the Toy interface for a robot, we would have to edit ToyHouse again to add another empty method, breaking open-closed principles.

10. Solution following ISP

To fix this, we segregate the bloated Toy interface into smaller "Role Interfaces". Classes then pick and choose the specific behaviors they need.

Good Design
interface Toy {
    void setPrice(double price);
    void setColor(String color);
}

interface Movable { void move(); }
interface Flyable { void fly(); }

// ToyHouse only implements Toy. No more dummy methods!
class ToyHouse implements Toy {
    double price; String color;
    public void setPrice(double p) { price = p; }
    public void setColor(String c) { color = c; }
}

// ToyPlane needs all behaviors, so it implements multiple interfaces!
class ToyPlane implements Toy, Movable, Flyable {
    double price; String color;
    public void setPrice(double p) { price = p; }
    public void setColor(String c) { color = c; }
    public void move(){ System.out.println("Moving"); }
    public void fly(){  System.out.println("Flying"); }
}

Key Takeaways

  • Role Interfaces: Break down monolithic interfaces into cohesive, focused units.
  • Composition over rigidity: Combine interfaces (implements Toy, Movable, Flyable) to construct complex class behaviors cleanly.
  • Code is now self-documenting and resilient to changes.

11. Finding the Architectural Balance

From a coding perspective, implementing an interface is incredibly simple—you just type the keyword implements. The real challenge is architectural.

1 Monolithic Interface
(Too rigid, dummy code)
Balanced Role Interfaces
(The Sweet Spot)
100 Tiny Interfaces
(Over-proliferation, messy)

Key Takeaways

  • Syntax is easy; architecture is hard. Knowing how to write implements is different from knowing when.
  • Avoid the extreme of creating single-method interfaces for every tiny thing. A pragmatic balance is required.
